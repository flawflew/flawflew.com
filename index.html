第一个需求开发总结

发表于 2022-01-23

第三周开始着手简单的API开发，只有简单的单表读写，也不涉及其他服务。本以为挺简单的（Controller->Service->Repository），结果花了一周都没搞定。

主要是还在模仿大家写的代码，这个过程中在设计的初衷，风格的选择上花了些时间。因为不了解促使这些组件或者设计方法产生的原因，会有：“这么做会不会是多此一举”，“会不会把问题复杂化了”等疑问。

主要的问题：

  • 代码主要是模仿为主，缺少了思考，面对多个已有的编码风格时，就不知所措了

    • 比如在Controller中去捕获异常是没有必要的，其实在com.xxx.xxx.xxx.configuration.WebMvcConfigurer中已经配置了统一异常解析器。

  • 在代码设计上缺少经验

    • 涉及到对象转换时，convert有必要用静态方法吗？convertTo还是convertFrom？Objects.nonNull 还是 obj != null？……

    • 会忽略掉潜在的NPE，意识不够

  • 定位解决问题很慢，很着急

    • 主要原因还是对组件和工具不了解

主要学习内容

项目流程

PM提出需求，需求评审，主R制定技术方案，技术方案评审，开发，联调，测试，上线。

上述只是一个大致的流程，其中每个环节都还包含许多细节。

  • 需求评审：由PM调研后提出需求，主要包含需求的背景，调研的数据，期望的结果，以及一些具体细节

  • 技术方案：由主R根据需求评审结果，制定出一个符合实际的可行方案，主要是使用UML描述核心技术细节：用例图，主要流程的时序图，API的初步定义……

  • 开发：参照技术方案，按照公司的开发规范具体实现。这里面研发有一定的自由度，但应该在整体上和团队编码风格和规范保持一致。与之并行的还有单元测试：在每完成一个小功能点时，应当进行测试验证，如果有上下游依赖可以模拟数据。

  • 联调：研发联系上下游的其他研发人员，共同测试整个链路的功能。遇到问题，共同协商解决。

  • 测试：完成联调后，在xxx上提测交付QA，由QA进行测试，这个过程中RD应尽力配合QA，时刻跟进解决发现的问题

    • 统一交付平台存在的意义：假如没有这个交付系统，就需要QA自行拉取代码，自行配置环境并构建后再测试。所以意义就是：使QA专注于测试，无需了解具体实现细节（个人理解）

  • 上线：在此之前，在仓库中创建release分支，项目进入staging阶段，将项目部署在线上环境，进行内部测试（类似于alpha测试吧），尽量模拟实际使用场景。之后进入prod阶段，即发布上线

    • 上线后应当检查：服务是否正常启动，端口是否正常，异常日志（使用跳板机登陆主机查看日志，或者在xxx上查看），还应当留意上下游是否出现异常

研发流程

首先，应当通过仔细的阅读需求文档，了解需求的背景和细节，针对疑问在需求评审中提出讨论，理解需求。

结合技术文档，分析是否满足需求，以及可行性。可以对技术文档做出优化。

在实际开发前，最好设计出关键数据的类图，关键流程的时序图。

开发内容主要分两个：

rpc服务

公司使用的是Thrift，首先根据上游业务需要，设计出服务接口。然后依照开发规范，实现接口。开发接口时，需要严格遵守API文档规范。在这次开发中也因为不规范导致服务异常，居然log只是打印了INFO级别的提示，导致找了很久原因。

http服务

根据技术文档中的API定义，依照开发规范设计接口。

其中服务注册和发现，路由，监控等基础的公共服务也是后端比较重要的内容，但是这些我感觉在开发过程中是完全透明的，只需要专注于写业务代码即可。

秋招简要总结

发表于 2021-12-11

秋招面试简要总结

故事背景：春招碰壁 + 考研就业间摇摆 + 秋招提前批已结束 + Java后端 + 0实习经验

先说点废话，本科其实学业并没有认真对待，绩点一度跌破3.0/5.0。但是，我可以拍着胸脯说，所有的课程设计和实验课我都认真对待了，也自认为属于动手能力较强的。花了很多时间整一些没啥技术力的玩意（主要在折腾那个白嫖的GCP的主机），现在看看就一 ScriptBoy。虽然没啥技术力，但是对于保持对计算机的热爱还是有帮助的。

简单说说春招，受挫的主要原因还是完全不了解面试流程 + 盲目自信，投了几家大厂，才知道人家根本不在乎我的那几个毫无特色课程设计，于是就只能八股文，而我完全没准备，但是好歹混了3年，其实大部分问题都有印象，只是无法给出一个自信的，标准的答案。然后就是G。现在想想挺后悔的，感觉这玩意会存有记录。

由于春招的失利，加上家里人（especially 我妈）一直希望我考研。然后我也临时的，短暂的加入了考研大军，看高数，看线代……直犯困，常常被B站首页推送的技术向视频吸引过去了。然后，在这学期开学前，我彻底放弃了。

关于秋招：

特别感谢我的同学 Emon ，其实在暑假他就提醒我提前批了，但是我没在意。开学后，就是完善简历，准备发言稿，八股文，复盘……每个环节他都给予了不少帮助。他还帮我内推了阿里和字节。字节没过，阿里倒在了HR面。主要原因还是准备不充分（按他的话说：我把自己给HR描述成了一个反人类不社交的Geeky Nerd），在阿里挂了HR面后开始认真复盘，每个环节都写逐字稿。之后拿了华为，美团还有一些非大厂Offer。

我觉得整个过程中最重要的部分有（分先后）：简历和发言稿，项目经历，八股文，手撕代码，其他。

  1. 简历和发言稿：这起到了引导面试官的作用，如果项目经历显得平平无奇，面试官没得问就只好八股文了。自信，只要是自己认真做了的项目，总会有亮点的。（我想到人工智能课的课设验收，介绍的环节，我就几句话说完了，一直强调没什么亮点，结果获得了第一名……都显得我虚伪了，后来想想其实我那段时间天天熬夜学习，虽然本质就是调参啦，但是里面有不少技巧使得整个过程高效很多）这里的发言稿不止自我介绍，还有应对常见问题的回答，尤其是HR面，你还没稳！！！
  2. 项目经历： 我重点介绍了我写的一个类UNIX文件系统，讲了如何解决并发，以及稳定性和效率间平衡问题。我没有参加过什么大创或者跟老师做项目，这是我最后一个全力以赴的课程设计。我认为比什么“购物网站”，“秒杀系统”等培训机构模板有吸引力。阿里和美团的面试官都对这个很感兴趣，美团二面甚至直接打开仓库看源码，完全就是我的主场了。
  3. 八股文： 我觉得临近突击一下可以，但是时间够多应该还是以系统学习为主。我学Java认真看了《深入理解JVM》，略看了《Thinking in Java》，八股文我觉得JavaGuide挺全面的了。
  4. 手撕代码： LeetCode 我做了12道，Nowcoder上做了29道，二者还有重叠的。所以，我没花太多精力在刷题上，优先级靠后。但是我面试中绝大多数都能ACC，我觉得这个环节应当多和面试官交流讨论。我有几次做不出来在和面试官讨论后发现是：描述有歧义，或者我想复杂了。
  5. 其他：比如能体现你计算机能力的兴趣爱好。我前面说的哪些没啥技术力的玩意还算有点帮助。华为的面试官认真和我讨论了《我如何“白嫖”校园网，并用路由器和室友共享》这件事。还有就是一些常见Linux问题，因为自己也时常使用，还算得心应手。

遗憾：

  1. 阿里HR面挂了，没有认真准备，在谈及缺点时，我有点过度自我贬低了（自卑人格？）。真的遗憾，二面面试官还约我吃饭来着。
  2. 很多公司没有面：腾讯，百度，快手，网易，京东……
  3. 没实习过……
