设计模式学习

发表于 2022-02-17 更新于 2022-03-02

创建型

简单工厂模式（静态工厂模式）

就像一家快餐店里的唯一一位厨师，菜单上的内容他都会。顾客并不在乎“安格斯牛堡”和“板烧鸡堡”到底怎么制作的，他只是想吃个汉堡而已，他要做的就是把“汉堡名字”告诉厨师就行了。

缺点：这位厨师必须样样都会，由于他的位置至关重要，所以也不能请假，出现新需求或者爆款时，厨师必须去学习。这样的工厂类太过于庞大复杂，扩展也麻烦，只要出现故障，整个系统都会出现问题。

使用场景：适合场景比较简单的情况，在JDK中也被广泛使用：java.txet.DateFormat.getInstance(); java.util.Calendar.getInstance();

工厂方法模式（虚拟构造器）

对于每个产品，都有一个专门负责创建的工厂类，就像其构造方法的作用一样，所以也称为虚拟构造器。这个模式克服了简单工厂复杂难扩展的问题。

缺点：每新增一个产品，都需要在引入一个专门的工厂类，系统复杂度增加。

为什么不直接使用构造器呢？我的理解是，顾客并不在乎“安格斯牛堡”的制作过程，甚至也不在乎“安格斯牛堡”本身，他只是饿了，想吃东西。

因为客户端依赖的仅仅是接口，所以可以很容易按需替换，类似Spring的依赖注入。比如我们都通过LogFactory来获取Log对象，但是最近Log4j出现严重问题，需要紧急替换，就可以使用配置文件加反射快速替换且无需更改原有代码。

行为型

观察者模式

Spring 中的观察者模式：

SpringBoot在启动过程中的各个关键事件都会触发相关监听器。org.springframework.boot.SpringApplication#run(java.lang.String…)

这些监听器是通过扫描所有autoconfigure包的META-INFO下的spring.factory的SpringApplicationRunListener引入的。
观察目标触发事件时，所有观察者都将被通知到，并根据通知作出反应，类似于“发布订阅”，是一种一对多的依赖关系。但观察者的变化无法被被观察者感知到。

适用情况：被观察目标需要通知其他对象，而不关心其数量或具体细节。

需要注意：目标与观察者之间不能循环依赖，可能会导致系统崩溃。观察者数量很多时，通知可能花费大量时间。

策略模式

Spring 中的策略模式：内容协商，http通信中客户端和服务端之间对于通信内容格式的协商

org.springframework.web.accept.ContentNegotiationManager#resolveMediaTypes，内容协商管理器根据请求体，解析出客户端接受的媒体类型。

这里会遍历strategy中的策略，但事实上SpringMVC默认只开启了HeaderContentNegotiationStrategy（根据请求头中的Accept解析的策略），还有多种策略可供选择，比如根据参数内容。

其实我不确定这算不算严格意义上的策略模式，也能时望文生义了。只是这段内容我印象比较深。除此之外在Bean实例化过程中也有策略模式，一种是适用JDK 反射生成代理，另一种是CGLIB，代码：org.springframework.beans.factory.support.SimpleInstantiationStrategy#instantiate(org.springframework.beans.factory.support.RootBeanDefinition, java.lang.String, org.springframework.beans.factory.BeanFactory)
策略模式的出发点，我认为，是对方法（或者算法）的封装，把相似作用的算法交给客户端，都他自己去选择使用那种策略。如果不使用策略模式，我们可能就只能通过硬编码，把各种算法写死在代码里，用冗长的条件判断去选择，当出现新的场景时，就不得不更改源码。

结构型

适配器模式

客户端需要使用符合Target接口的对象，但是当前服务并不满足去需求。这时候，我们可以选择去改动源码以符合去需求，但是如果需求变化频繁，这将是个头疼的问题。举例：Macbook 被设计工作在20V 3A的直流电源下，但是全球各国的电源标准各不相同，且通常都是交流电。我们当然可以做到：重新设计入户电源，使其符合Mac标准，但是家里不只又Mac还有电视，空调，冰箱……对于电源这种被大量需求的对象，显然是不宜对其做出变动的，所以Macbook根据地区标准出厂即配备了电源适配器。

这里涉及两方：客户端（使用服务），服务端（提供服务）

问题在于二者没法协调，如何解决？

- 客户端做出妥协，主动配合
- 服务端做出妥协
- 在二者之间配个“翻译”

适配器模式属于第3种，客户端，服务端都不需要做出改变，这样对整个系统的影响最小。

思考：如果把“适配器”用胶带绑在Mac上，或者直接把电路设计到Mac的主板里，成为Mac的一部分，这样是不是看起来“更简洁”？这是不是就变成了第1种思路，客户端做出改变？标准答案可能是：这样降低了MacBook整体的功能内聚度，把功能差异很大的组件耦合在了一块。在做系统设计时，该怎么划分模块间的界限呢？以何种粒度分割？我觉得这很难有标准答案，不同的方案各有利弊，需要结合实际项目加以权衡。

装饰模式

假如现在要对一个类的功能进行扩展，我们第一直觉告诉我们：继承，这也符合 “extends” 的本意。确实这么做可以满足绝大多数场景，但同时存在一些问题。（找一个合适的例子总是很困难）

快餐店里有鸡腿堡，牛堡，虾堡……M种汉堡，根据顾客的口味喜好，允许顾客提出定制化要求，比如：添加腌黄瓜，洋葱，生菜……。如果按照继承的思路，那么，就需要对每种汉堡都重新设计配方（生成子类），对于“腌黄瓜”一种选项，凭空就多了M个汉堡配方。但是除了腌黄瓜，还有洋葱，生菜，番茄，芝士，培根等N种不同选择，这下，我们就多出了M*N种配方，而且还允许各种搭配，各种数量，这下越来越难以管理了。其实，我们知道，“加一片芝士”这个操作与汉堡种类无关，只要是个汉堡就行了，收到顾客需求，我们只需要添加一份备忘录：顾客A需要额外加片培根。就完事了。也解释装饰器，因为装饰器本身也是Component的子类，所以我们可以多个装饰器“串联”起来，实现不同搭配。

装饰模式可以更加灵活的扩展，可以避免继承滥用，同时一些被final修饰的，无法继承的类，可以使用类似的思想去增强。

代理模式

看上去，代理模式和装饰模式有些像，都好像是一个Wrapper，把真实对象包装起来。但二者的动机是不一样的。不仅仅是在代码中，“代理”一词在生活中也很常见，代理人，代理商……顾名思义，之所以要人代理，是因为本人不方便出面，所以找一个信得过的人代为处理，同时，不会对结果产生本质改变。在应用层面，代理分为正向代理和反向代理。正向代理：代理客户端，向服务端隐藏细节；反向代理：代理服务端，向客户端隐藏细节。前者常用在客户端网络代理上，后者则常用在服务端负载均衡上。代码层面，代理又有动态静态之分，前者类似装饰模式，需要手动定义代理对象，后者则是依靠API自动在内存中构造，具有更高的灵活性。在Spring AOP中也常见代理模式，主要有两种一种是基于JDK的动态代理，另一种是基于CGLIB。在我们debug时，可以看到有的Bean标注了EnhancerByCglib。二者的区别在于JDK代理的对象必须要基于某个接口，而CGLIB不用，CGLIB是通过编辑class字节码+反射生成代理对象的。

第一个需求开发总结

发表于 2022-01-23

第三周开始着手简单的API开发，只有简单的单表读写，也不涉及其他服务。本以为挺简单的（Controller->Service->Repository），结果花了一周都没搞定。

主要是还在模仿大家写的代码，这个过程中在设计的初衷，风格的选择上花了些时间。因为不了解促使这些组件或者设计方法产生的原因，会有：“这么做会不会是多此一举”，“会不会把问题复杂化了”等疑问。

主要的问题：

  • 代码主要是模仿为主，缺少了思考，面对多个已有的编码风格时，就不知所措了

    • 比如在Controller中去捕获异常是没有必要的，其实在com.xxx.xxx.xxx.configuration.WebMvcConfigurer中已经配置了统一异常解析器。

  • 在代码设计上缺少经验

    • 涉及到对象转换时，convert有必要用静态方法吗？convertTo还是convertFrom？Objects.nonNull 还是 obj != null？……

    • 会忽略掉潜在的NPE，意识不够

  • 定位解决问题很慢，很着急

    • 主要原因还是对组件和工具不了解

主要学习内容

项目流程

PM提出需求，需求评审，主R制定技术方案，技术方案评审，开发，联调，测试，上线。

上述只是一个大致的流程，其中每个环节都还包含许多细节。

  • 需求评审：由PM调研后提出需求，主要包含需求的背景，调研的数据，期望的结果，以及一些具体细节

  • 技术方案：由主R根据需求评审结果，制定出一个符合实际的可行方案，主要是使用UML描述核心技术细节：用例图，主要流程的时序图，API的初步定义……

  • 开发：参照技术方案，按照公司的开发规范具体实现。这里面研发有一定的自由度，但应该在整体上和团队编码风格和规范保持一致。与之并行的还有单元测试：在每完成一个小功能点时，应当进行测试验证，如果有上下游依赖可以模拟数据。

  • 联调：研发联系上下游的其他研发人员，共同测试整个链路的功能。遇到问题，共同协商解决。

  • 测试：完成联调后，在xxx上提测交付QA，由QA进行测试，这个过程中RD应尽力配合QA，时刻跟进解决发现的问题

    • 统一交付平台存在的意义：假如没有这个交付系统，就需要QA自行拉取代码，自行配置环境并构建后再测试。所以意义就是：使QA专注于测试，无需了解具体实现细节（个人理解）

  • 上线：在此之前，在仓库中创建release分支，项目进入staging阶段，将项目部署在线上环境，进行内部测试（类似于alpha测试吧），尽量模拟实际使用场景。之后进入prod阶段，即发布上线

    • 上线后应当检查：服务是否正常启动，端口是否正常，异常日志（使用跳板机登陆主机查看日志，或者在xxx上查看），还应当留意上下游是否出现异常

研发流程

首先，应当通过仔细的阅读需求文档，了解需求的背景和细节，针对疑问在需求评审中提出讨论，理解需求。

结合技术文档，分析是否满足需求，以及可行性。可以对技术文档做出优化。

在实际开发前，最好设计出关键数据的类图，关键流程的时序图。

开发内容主要分两个：

rpc服务

公司使用的是Thrift，首先根据上游业务需要，设计出服务接口。然后依照开发规范，实现接口。开发接口时，需要严格遵守API文档规范。在这次开发中也因为不规范导致服务异常，居然log只是打印了INFO级别的提示，导致找了很久原因。

http服务

根据技术文档中的API定义，依照开发规范设计接口。

其中服务注册和发现，路由，监控等基础的公共服务也是后端比较重要的内容，但是这些我感觉在开发过程中是完全透明的，只需要专注于写业务代码即可。

秋招简要总结

发表于 2021-12-11

秋招面试简要总结

故事背景：春招碰壁 + 考研就业间摇摆 + 秋招提前批已结束 + Java后端 + 0实习经验

先说点废话，本科其实学业并没有认真对待，绩点一度跌破3.0/5.0。但是，我可以拍着胸脯说，所有的课程设计和实验课我都认真对待了，也自认为属于动手能力较强的。花了很多时间整一些没啥技术力的玩意（主要在折腾那个白嫖的GCP的主机），现在看看就一 ScriptBoy。虽然没啥技术力，但是对于保持对计算机的热爱还是有帮助的。

简单说说春招，受挫的主要原因还是完全不了解面试流程 + 盲目自信，投了几家大厂，才知道人家根本不在乎我的那几个毫无特色课程设计，于是就只能八股文，而我完全没准备，但是好歹混了3年，其实大部分问题都有印象，只是无法给出一个自信的，标准的答案。然后就是G。现在想想挺后悔的，感觉这玩意会存有记录。

由于春招的失利，加上家里人（especially 我妈）一直希望我考研。然后我也临时的，短暂的加入了考研大军，看高数，看线代……直犯困，常常被B站首页推送的技术向视频吸引过去了。然后，在这学期开学前，我彻底放弃了。

关于秋招：

特别感谢我的同学 Emon ，其实在暑假他就提醒我提前批了，但是我没在意。开学后，就是完善简历，准备发言稿，八股文，复盘……每个环节他都给予了不少帮助。他还帮我内推了阿里和字节。字节没过，阿里倒在了HR面。主要原因还是准备不充分（按他的话说：我把自己给HR描述成了一个反人类不社交的Geeky Nerd），在阿里挂了HR面后开始认真复盘，每个环节都写逐字稿。之后拿了华为，美团还有一些非大厂Offer。

我觉得整个过程中最重要的部分有（分先后）：简历和发言稿，项目经历，八股文，手撕代码，其他。

  1. 简历和发言稿：这起到了引导面试官的作用，如果项目经历显得平平无奇，面试官没得问就只好八股文了。自信，只要是自己认真做了的项目，总会有亮点的。（我想到人工智能课的课设验收，介绍的环节，我就几句话说完了，一直强调没什么亮点，结果获得了第一名……都显得我虚伪了，后来想想其实我那段时间天天熬夜学习，虽然本质就是调参啦，但是里面有不少技巧使得整个过程高效很多）这里的发言稿不止自我介绍，还有应对常见问题的回答，尤其是HR面，你还没稳！！！
  2. 项目经历： 我重点介绍了我写的一个类UNIX文件系统，讲了如何解决并发，以及稳定性和效率间平衡问题。我没有参加过什么大创或者跟老师做项目，这是我最后一个全力以赴的课程设计。我认为比什么“购物网站”，“秒杀系统”等培训机构模板有吸引力。阿里和美团的面试官都对这个很感兴趣，美团二面甚至直接打开仓库看源码，完全就是我的主场了。
  3. 八股文： 我觉得临近突击一下可以，但是时间够多应该还是以系统学习为主。我学Java认真看了《深入理解JVM》，略看了《Thinking in Java》，八股文我觉得JavaGuide挺全面的了。
  4. 手撕代码： LeetCode 我做了12道，Nowcoder上做了29道，二者还有重叠的。所以，我没花太多精力在刷题上，优先级靠后。但是我面试中绝大多数都能ACC，我觉得这个环节应当多和面试官交流讨论。我有几次做不出来在和面试官讨论后发现是：描述有歧义，或者我想复杂了。
  5. 其他：比如能体现你计算机能力的兴趣爱好。我前面说的哪些没啥技术力的玩意还算有点帮助。华为的面试官认真和我讨论了《我如何“白嫖”校园网，并用路由器和室友共享》这件事。还有就是一些常见Linux问题，因为自己也时常使用，还算得心应手。

遗憾：

  1. 阿里HR面挂了，没有认真准备，在谈及缺点时，我有点过度自我贬低了（自卑人格？）。真的遗憾，二面面试官还约我吃饭来着。
  2. 很多公司没有面：腾讯，百度，快手，网易，京东……
  3. 没实习过……
